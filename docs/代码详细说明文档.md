# 深度调研智能体 — 代码详细说明文档

## 一、项目概述

**深度调研智能体（Deep Research Agent）** 是一个基于 Web 的自主 AI 调研系统。用户输入研究问题后，Agent 通过 **ReAct（Thought-Action-Observation）循环** 自主完成搜索、阅读、分析，最终生成带有来源引用的 Markdown 调研报告。所有步骤通过 SSE（Server-Sent Events）实时推送到前端展示。Agent 使用 OpenAI **Function Calling**（工具调用）机制实现结构化的工具选择和执行，替代了传统的文本解析方式。

**核心特点：**
- 完全不依赖 LangChain、LlamaIndex 等第三方 Agent 框架，核心模块全部自研
- 支持任意 OpenAI 兼容接口的 LLM（当前配置为 DeepSeek Chat）
- 前端 Vue 3 + 后端 FastAPI，全栈分离架构
- 全中文化：UI、System Prompt、工具描述、错误信息均为中文

**技术栈：**

| 层 | 选型 |
|----|------|
| 后端框架 | FastAPI (Python) |
| 前端 | Vue 3 (Vite) |
| LLM API | OpenAI 兼容接口 |
| 搜索 API | SerpAPI (Google Search) |
| 网页抓取 | httpx + BeautifulSoup4 |
| 数据库 | SQLite + aiosqlite (异步) |
| 前端 Markdown | marked.js |

## 三、Agent 运行流程（完整过程）

整个 Agent 的调研过程分为 **三个阶段**，由 `ResearchOrchestrator` 统一编排。从用户输入问题到最终报告生成，完整流程如下：

```
用户输入问题
    │
    ▼
┌──────────────────────────────────────────────────┐
│  阶段一：生成调研计划                              │
│  ResearchOrchestrator.research()                 │
│  └─ LLMClient.chat_json_mode(query)              │
│     → 将问题拆分为 2-4 个子问题（JSON Mode）        │
│     → 通过 SSE 发送 event: plan                   │
└──────────────┬───────────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────────┐
│  阶段二：ReAct 循环执行（核心）                     │
│  AgentLoop.run(query)                            │
│                                                  │
│  循环（最多 10 轮）：                              │
│    ┌──────────────────────┐                      │
│    │ 1. Function Calling   │  ← 发送 tools 定义    │
│    │    LLM 返回 content + │                      │
│    │    tool_calls 或纯文本 │                      │
│    └─────────┬────────────┘                      │
│              │                                    │
│              ▼                                    │
│    ┌──────────────────────┐                      │
│    │ 2. 提取 Thought       │  ← content 即为思考   │
│    │    content → thought  │    (无解析器, LLM     │
│    │    原生支持结构输出    │     自行决定格式)      │
│    └─────────┬────────────┘                      │
│              │                                    │
│              ▼                                    │
│    ┌──────────────────────┐                      │
│    │ 3. 执行工具调用        │  ← ToolRegistry       │
│    │    web_search()       │    tool_calls[] 中    │
│    │    fetch_page()       │    提取函数名和参数    │
│    │    submit_report()    │                      │
│    └─────────┬────────────┘                      │
│              │                                    │
│              ▼                                    │
│    ┌──────────────────────┐                      │
│    │ 4. 记录观察 + 记忆管理 │  ← MemoryManager     │
│    │    存入 Observation   │                      │
│    │    提取关键事实        │                      │
│    │    超限时压缩历史      │                      │
│    └─────────┬────────────┘                      │
│              │                                    │
│              ▼                                    │
│    ┌──────────────────────┐                      │
│    │ 5. 判断终止条件        │                      │
│    │    submit_report? → 退出                      │
│    │    达到 MAX_ROUNDS?  → 强制摘要               │
│    │    否则 → 回到步骤 1   │                      │
│    └──────────────────────┘                      │
│                                                  │
│  每步通过 SSE 发送 event: step                     │
└──────────────┬───────────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────────┐
│  阶段三：生成最终报告                              │
│  报告由 AgentLoop 自动生成                        │
│  └─ submit_report 工具调用时，LLM 已输出完整报告    │
│  └─ loop.final_report 包含 Markdown 报告内容       │
│  └─ 提取来源 URL 列表                             │
│  └─ 通过 SSE 发送 event: report                   │
└──────────────────────────────────────────────────┘
```

### 4.4 LLM 客户端 — `backend/llm/client.py`

```python
# 文件：backend/llm/client.py
```

**职责：** 封装与 OpenAI 兼容 API 的通信，提供三个核心方法。

**核心方法：**

#### `chat(messages, max_tokens=4096) → str`
通用对话方法。使用 `temperature=0.3`（低温度保证输出稳定性），发送消息列表并返回 LLM 的文本响应。

#### `chat_json_mode(messages, max_tokens=4096) → dict`
**JSON Mode 调用。** 使用 `response_format={"type": "json_object"}` 要求 LLM 返回结构化 JSON。将响应解析为 Python dict 并返回。JSON 解析失败时返回空字典。用于计划生成等需要结构化输出的场景。

#### `chat_with_tools(messages, tools, max_tokens=4096) → dict`
**Function Calling 调用。** 使用 OpenAI 原生 `tools` 参数发送工具定义，LLM 可自行决定是否调用工具。返回 dict 包含：
- `role`: 消息角色（assistant）
- `content`: 文本内容（思考/回答）
- `tool_calls`: 工具调用列表（如有），每个包含 `id`, `name`, `arguments`

#### ~~`generate_plan(query) → list[str]`~~（已移除）
被 `chat_json_mode()` 替代。Orchestrator 中计划生成改为调用 `chat_json_mode()` + prompt，不再需要专门的 `generate_plan` 方法。

**设计要点：**
- 使用 `AsyncOpenAI` 客户端，所有方法均为异步
- `base_url` 可配置，支持任何 OpenAI 兼容服务（如 DeepSeek、本地 Ollama 等）
- 模型名称可通过 `.env` 灵活切换
- 新增的 `chat_json_mode()` 和 `chat_with_tools()` 利用了 OpenAI API 原生能力，可靠性和可维护性远超文本解析方案

### 4.5 Agent 核心引擎

#### 4.5.1 调研编排器 — `backend/agent/orchestrator.py`

```python
# 文件：backend/agent/orchestrator.py
```

**职责：** 作为最高层控制器，编排整个调研流程的三个阶段。

**核心类：`ResearchOrchestrator`**

**`__init__()`**：创建 `LLMClient` 实例和 `ToolRegistry` 实例，通过 `_setup_tools()` 注册三个内置工具。

**`_setup_tools()`**：向注册中心注册三个工具：

| 工具名 | 描述 | 参数 | 处理函数 |
|--------|------|------|----------|
| `web_search` | 在互联网上搜索信息 | `query: string` | `web_search()` |
| `fetch_page` | 抓取并提取指定 URL 的文本内容 | `url: string` | `fetch_page()` |
| `submit_report` | 提交最终调研报告 | `report: string` | 标记循环结束 |

**`research(query) → AsyncGenerator[dict, None]`**：核心调研流程，是一个异步生成器，yield 的事件通过 SSE 实时推送。

**三阶段流程：**

1. **生成计划**：调用 `self.llm.chat_json_mode(messages)`（使用 JSON Mode），将问题拆分为 2-4 个子问题，yield `PlanEvent`
2. **ReAct 循环**：创建 `AgentLoop` 实例，通过 `self.registry.get_openai_tools()` 获取 tools 定义传入 `loop.run(query, tool_defs=tools)`，逐步骤 yield `StepEvent`
3. **生成报告**：循环结束后，`loop.final_report` 包含 submit_report 工具调用时 LLM 输出的完整 Markdown 报告。通过正则提取所有 URL 作为来源列表，yield `ReportEvent`

**`_extract_sources(report) → list[str]`**：用正则 `https?://[^\s\)\]]+` 从报告中提取所有 URL，并用 `dict.fromkeys()` 去重保留顺序。

#### 4.5.2 ReAct 循环引擎 — `backend/agent/loop.py`

```python
# 文件：backend/agent/loop.py
```

**职责：** 实现 ReAct（Reasoning + Acting）范式的核心循环，是 Agent 的"大脑"。使用 OpenAI **Function Calling** 机制实现工具调用，不再依赖文本解析。

**System Prompt（中文）：**

```
你是一个深度调研智能体。你的目标是对研究主题进行彻底调查，
生成一份全面、来源可靠的答案。

你可以使用提供的工具进行搜索和阅读。请逐步深入调研，
当你认为信息足够全面时，使用 submit_report 工具提交最终报告。

规则：
- 先搜索后阅读：先用 web_search 搜索，再对最有价值的 URL 使用 fetch_page 深入阅读
- 多源验证：重要结论不能只依赖单一来源
- 引用来源格式：[来源: URL]
- 如果搜索失败或无结果，调整策略或诚实地报告未能找到的内容
```

**核心类：`AgentLoop`**

**`__init__(tool_registry, llm)`**：接收工具注册中心和 LLM 客户端，创建 `MemoryManager` 实例。

**`run(query, tool_defs=None) → AsyncGenerator[StepEvent, None]`**：ReAct 主循环，现已接受 `tool_defs` 参数（OpenAI tools 格式的工具定义）。

**循环逻辑（`for round_num in range(1, MAX_ROUNDS + 1)`）：**

1. **Function Calling**：调用 `self.llm.chat_with_tools(messages, tools=tool_defs)` 获取 LLM 响应。LLM 返回：
   - `content`：思考文本（yield 为 thought StepEvent）
   - `tool_calls`：工具调用列表（如有），每个包含 `id`, `name`, `arguments`
2. **无工具调用**：如果 `tool_calls` 为空，表示 LLM 没有调用任何工具的意图，结束循环
3. **执行工具**：遍历 `tool_calls`：
   - 调用 `self.tool_registry.execute(tool_name, tool_args)` 执行工具
   - 发送 `action` 类型的 StepEvent（工具名和参数）
   - 发送 `observation` 类型的 StepEvent（内容截断为 2000 字）
   - 将 assistant 响应和 tool result 消息追加到 messages 上下文
   - 如果 Observation 超过 200 字，提取前 300 字作为关键事实存入 MemoryManager
4. **终止判断**：如果 tool_calls 中包含 `submit_report`，将报告内容存入 `self.final_report`，结束循环并 `return`
5. **强制终止**：达到 MAX_ROUNDS 后，调用 LLM 基于 `memory.get_steps_for_context()` 生成摘要报告

**设计要点：**
- 使用 OpenAI 原生 Function Calling，可靠性和结构化程度远超文本解析
- 每轮都将完整的 `messages` 历史发送给 LLM（包括 tool role 消息），保证上下文连贯
- Observation 内容截断为 2000 字，防止单轮上下文爆炸
- 只有 Observation 长度 > 200 时才提取关键事实（过滤掉无意义的短结果）
- `submit_report` 工具作为循环的优雅终止方式，替代了旧版 FINAL_ANSWER 关键字检测
- 达到最大轮数时强制生成摘要，防止无限循环

#### 4.5.3 输出解析器 — ~~`backend/agent/parser.py`~~（已删除）

> **状态：已删除。改为使用 OpenAI Function Calling。**

旧版 Agent 使用 `parser.py` 从 LLM 的原始文本输出中提取 `THOUGHT`、`ACTION`（`工具名|JSON参数`）和 `FINAL_ANSWER`。这是基于正则表达式的文本解析方案，需要处理 LLM 格式不稳定的问题。

#### 4.5.4 记忆管理器 — `backend/agent/memory.py`

```python
# 文件：backend/agent/memory.py
```

**职责：** 管理 Agent 的步骤历史，在上下文过长时自动压缩，同时保留关键事实。

**核心类：`MemoryManager`**

**属性：**
| 属性 | 类型 | 说明 |
|------|------|------|
| `steps` | `list[Step]` | 完整的步骤历史记录 |
| `key_facts` | `list[str]` | 去重的关键事实列表 |
| `max_history_chars` | `int` | 压缩阈值，默认 8000 字符 |

**方法：**

**`add_step(step)`**：将步骤追加到 `steps` 列表。

**`add_key_fact(fact)`**：将关键事实追加到 `key_facts` 列表（自动去重）。在 `loop.py` 中，只有 Observation 长度 > 200 时才提取前 300 字作为关键事实。

**`get_steps_for_context() → str`**：构建用于发送给 LLM 的上下文字符串。

- **正常模式**：如果所有步骤的格式化字符串总长度 ≤ `max_history_chars`，直接拼接返回
- **压缩模式**：如果超出限制：
  1. 从后往前取最近的步骤，直到达到 `max_history_chars // 2`（保留最近 4000 字符）
  2. 对前面的旧步骤调用 `_build_summary()` 生成压缩摘要
  3. 返回格式：`[摘要] + --- Recent Steps --- + [最近步骤]`

**`_build_summary(early_steps)`**：生成压缩摘要，包含：
- 被压缩的步骤数量
- 早期执行的操作数量
- 所有已收集的关键事实列表

**设计要点：**
- 压缩策略：保留最近步骤的完整内容（因为最新信息最重要），旧步骤只保留摘要
- 关键事实独立于步骤存储，即使步骤被压缩，关键事实也不会丢失
- 500 字符的关键事实提取阈值有效过滤了空结果和错误信息

#### 4.5.5 工具注册中心 — `backend/agent/tools/registry.py`

```python
# 文件：backend/agent/tools/registry.py
```

**职责：** 提供插件式的工具注册和执行机制，使工具的管理与 Agent 核心逻辑解耦。

**`Tool` 数据类：**

```python

    name: str           # 工具名称（LLM 通过名称调用）
    description: str    # 工具描述（注入到 System Prompt）
    parameters: dict    # 参数 JSON Schema（注入到 System Prompt）
    handler: Callable   # 实际执行函数
```

**`ToolRegistry` 类：**

| 方法 | 说明 |
|------|------|
| `register(tool)` | 注册工具到 `_tools` 字典 |
| `get(name)` | 按名称获取工具 |
| `get_openai_tools()` | 生成 OpenAI Function Calling 格式的工具定义列表 |
| `execute(name, params)` | 按名称查找工具并执行，返回字符串结果 |

**`get_openai_tools()` 替换了旧版 `get_schema_for_llm()`**：不再生成文本格式的工具描述注入 System Prompt，而是生成 OpenAI Function Calling 所需的 `tools` 参数格式（`[{"type": "function", "function": {"name": "...", "description": "...", "parameters": {...}}}]`），直接传给 `chat_with_tools()`。

**`execute()` 的错误处理：**
- 工具未找到：返回 `错误：工具 'xxx' 未找到`
- 执行异常：返回 `执行 'xxx' 出错：{异常信息}`

所有异常都被捕获并返回为字符串，确保 Agent 循环不会因工具执行失败而崩溃。LLM 可以根据错误信息调整策略。

#### 4.5.6 Web 搜索工具 — `backend/agent/tools/web_search.py`

```python
# 文件：backend/agent/tools/web_search.py
```

**函数签名：** `async web_search(query: str, num: int = 5) → str`

**执行流程：**
1. 检查 `SEARCH_API_KEY` 是否配置，未配置则返回提示信息

- API Key 未配置：返回提示信息
- 请求超时（`httpx.TimeoutException`）：返回超时提示，建议使用更精确的搜索词
- 其他异常：返回失败信息和异常详情
- 无搜索结果：返回 `未找到与「xxx」相关的结果`

#### 4.5.7 网页抓取工具 — `backend/agent/tools/fetch_page.py`

```python
# 文件：backend/agent/tools/fetch_page.py
```

**函数签名：** `async fetch_page(url: str, max_chars: int = 5000) → str`

**执行流程：**
1. 使用 `httpx.AsyncClient`（20 秒超时，因为网页可能较大）发送 GET 请求
2. 设置 User-Agent 为 `Mozilla/5.0 (compatible; ResearchAgent/1.0)`，声明身份
3. 使用 BeautifulSoup 解析 HTML
4. **清洗 HTML**：删除 `script`、`style`、`nav`、`footer`、`header`、`aside` 标签（这些通常不是正文内容）
5. 使用 `get_text(separator="\n", strip=True)` 提取纯文本
6. 去除空行，将文本拼接
7. 如果超过 `max_chars`（5000 字），截断并标注 `[内容已截断，超过 5000 字符]`
8. 提取网页标题，返回格式：`标题：...\n链接：...\n\n{正文}`

**错误处理：**
- 超时：返回 `抓取 url 超时，将使用搜索摘要`
- 其他异常：返回 `抓取 url 失败：{异常}，将使用搜索摘要`

异常信息中的"将使用搜索摘要"提示 LLM 可以依赖已有的搜索摘要继续工作。

#### `POST /api/research` — 发起调研（SSE 流）

- 接收 `{"query": "..."}` 请求体

#### 4.8.2 SSE 流式接收 — `frontend/src/composables/useSSE.js`

**职责：** 封装基于 Fetch API 的 SSE 流式接收逻辑。**不使用浏览器原生 `EventSource`**，因为调研接口是 POST 请求（EventSource 仅支持 GET）。

**`startResearch(query)` 核心流程：**

1. 重置所有响应式状态

| `plan` | 更新 `subQuestions` |

## 六、容错与异常处理

| 场景 | 位置 | 处理策略 |
|------|------|----------|
| Function Calling 无工具调用 | `loop.py` | 无 tool_calls 时自动结束循环 |
| Search API 超时/限流 | `web_search.py` | 返回友好提示，LLM 可基于已有信息继续或调整搜索词 |
| 网页抓取失败 | `fetch_page.py` | 降级为"使用搜索摘要"，标注未能获取全文 |
| LLM 不调用 submit_report | `loop.py` | 硬上限 10 轮后强制调用 LLM 生成摘要报告 |
| 上下文超限（>8000字） | `memory.py` | 压缩早期步骤为摘要，保留最近步骤完整内容 |
| 用户关闭浏览器/断开 SSE | `routes/research.py` | 已完成步骤已实时写入数据库（收集后统一更新） |
| 工具执行异常 | `registry.py` | 捕获所有异常并返回字符串错误信息，不抛异常 |
| JSON 解析失败（plan） | `client.py` | chat_json_mode 解析失败时返回空字典降级 |
| 数据库连接异常 | `connection.py` | 应用关闭时通过 lifespan 确保连接正确关闭 |

## 八、关键设计决策

1. **不使用 Agent 框架**：LangChain/LlamaIndex 增加了抽象层级和依赖复杂度，对于 ReAct 这种简单范式，自研约 500 行代码即可完整实现，且完全可控。迁移到 Function Calling 后代码更简洁——移除了 parser.py 和 cross_check.py，Agent 核心逻辑约 450 行。

2. **OpenAI Function Calling 而非文本解析**：早期版本使用正则解析 LLM 文本输出（THOUGHT/ACTION/FINAL_ANSWER），存在格式不稳定风险。迁移到 OpenAI 原生 Function Calling 后，工具调用完全结构化，可靠性从约 95% 提升到接近 100%，同时移除了整个解析器模块。

3. **SSE 而非 WebSocket**：调研场景是单向的服务器→客户端推送，SSE 比 WebSocket 更轻量，浏览器原生支持自动重连，且兼容 HTTP/2。

4. **Fetch API 而非 EventSource**：因为需要 POST 请求发送查询内容，EventSource 仅支持 GET，所以使用 Fetch API + ReadableStream 手动解析 SSE 流。

5. **低温度参数 (0.3)**：调研任务需要事实准确性而非创意性，低温度减少 LLM 的随机性，提高输出的一致性和可靠性。

6. **先压缩后截断**：MemoryManager 不是简单的字符截断，而是先压缩旧步骤为摘要再拼接最近步骤，保留最重要的上下文信息。

7. **所有错误都返回字符串**：Tool 执行失败时不抛异常，而是返回字符串错误信息，让 LLM 可以"读到"错误并据此调整策略，这是 ReAct 范式的核心优势。

8. **WAL 模式 SQLite**：Write-Ahead Logging 允许多个读操作与一个写操作并发执行，对 SSE 流式写入场景友好。
