# 深度调研智能体 — 代码详细说明文档

## 一、项目概述

**深度调研智能体（Deep Research Agent）** 是一个基于 Web 的自主 AI 调研系统。用户输入研究问题后，Agent 通过 **ReAct（Thought-Action-Observation）循环** 自主完成搜索、阅读、分析、交叉验证，最终生成带有来源引用的 Markdown 调研报告。所有步骤通过 SSE（Server-Sent Events）实时推送到前端展示。

**核心特点：**
- 完全不依赖 LangChain、LlamaIndex 等第三方 Agent 框架，核心模块全部自研
- 支持任意 OpenAI 兼容接口的 LLM（当前配置为 DeepSeek Chat）
- 前端 Vue 3 + 后端 FastAPI，全栈分离架构
- 全中文化：UI、System Prompt、工具描述、错误信息均为中文

**技术栈：**

| 层 | 选型 |
|----|------|
| 后端框架 | FastAPI (Python) |
| 前端 | Vue 3 (Vite) |
| LLM API | OpenAI 兼容接口 |
| 搜索 API | SerpAPI (Google Search) |
| 网页抓取 | httpx + BeautifulSoup4 |
| 数据库 | SQLite + aiosqlite (异步) |
| 前端 Markdown | marked.js |

## 三、Agent 运行流程（完整过程）

整个 Agent 的调研过程分为 **四个阶段**，由 `ResearchOrchestrator` 统一编排。从用户输入问题到最终报告生成，完整流程如下：

```
用户输入问题
    │
    ▼
┌──────────────────────────────────────────────────┐
│  阶段一：生成调研计划                              │
│  ResearchOrchestrator.research()                 │
│  └─ LLMClient.generate_plan(query)               │
│     → 将问题拆分为 2-4 个子问题                    │
│     → 通过 SSE 发送 event: plan                   │
└──────────────┬───────────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────────┐
│  阶段二：ReAct 循环执行（核心）                     │
│  AgentLoop.run(query)                            │
│                                                  │
│  循环（最多 10 轮）：                              │
│    ┌──────────────────────┐                      │
│    │ 1. LLM 推理           │  ← 发送完整上下文      │
│    │    生成 Thought +     │                      │
│    │    Action/FinalAnswer │                      │
│    └─────────┬────────────┘                      │
│              │                                    │
│              ▼                                    │
│    ┌──────────────────────┐                      │
│    │ 2. Parser 解析        │  ← 提取结构化指令      │
│    │    解析 THOUGHT       │                      │
│    │    解析 ACTION        │  解析失败? → 重试一次  │
│    │    识别 FINAL_ANSWER  │                      │
│    └─────────┬────────────┘                      │
│              │                                    │
│              ▼                                    │
│    ┌──────────────────────┐                      │
│    │ 3. 执行工具           │  ← ToolRegistry       │
│    │    web_search()       │                      │
│    │    fetch_page()       │                      │
│    │    cross_check()      │                      │
│    └─────────┬────────────┘                      │
│              │                                    │
│              ▼                                    │
│    ┌──────────────────────┐                      │
│    │ 4. 记录观察 + 记忆管理 │  ← MemoryManager     │
│    │    存入 Observation   │                      │
│    │    提取关键事实        │                      │
│    │    超限时压缩历史      │                      │
│    └─────────┬────────────┘                      │
│              │                                    │
│              ▼                                    │
│    ┌──────────────────────┐                      │
│    │ 5. 判断终止条件        │                      │
│    │    FINAL_ANSWER? → 退出                       │
│    │    达到 MAX_ROUNDS?  → 强制摘要               │
│    │    否则 → 回到步骤 1   │                      │
│    └──────────────────────┘                      │
│                                                  │
│  每步通过 SSE 发送 event: step                     │
└──────────────┬───────────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────────┐
│  阶段三：交叉验证                                  │
│  cross_check(facts)                              │
│  └─ 取最近 5 条 Observation                       │
│  └─ 调用 LLM 检查一致性、矛盾、已验证事实            │
│  └─ 通过 SSE 发送 event: cross_check              │
└──────────────┬───────────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────────┐
│  阶段四：生成最终报告                              │
│  LLMClient.chat()                                │
│  └─ 汇总所有信息 + 交叉验证结果                     │
│  └─ 生成 Markdown 格式报告（含内联引用）            │
│  └─ 提取来源 URL 列表                             │
│  └─ 通过 SSE 发送 event: report                   │
└──────────────────────────────────────────────────┘
```

### 4.4 LLM 客户端 — `backend/llm/client.py`

```python
# 文件：backend/llm/client.py
```

**职责：** 封装与 OpenAI 兼容 API 的通信，提供三个核心方法。

**核心方法：**

#### `chat(messages, max_tokens=4096) → str`
通用对话方法。使用 `temperature=0.3`（低温度保证输出稳定性），发送消息列表并返回 LLM 的文本响应。所有 LLM 调用最终都通过此方法。

#### `generate_plan(query) → list[str]`
生成调研计划。发送特定 prompt 要求 LLM 将问题拆分为 2-4 个子问题，返回 JSON 字符串数组。如果 JSON 解析失败，降级为返回 `[原问题]`。

```python
# prompt 设计：
"将以下研究问题拆分为 2-4 个具体的子问题。只返回一个 JSON 字符串数组，不要包含其他内容。"
```

#### `summarize(text, max_length=500) → str`
文本摘要方法。要求 LLM 将输入文本压缩到指定字数以内。

**设计要点：**
- 使用 `AsyncOpenAI` 客户端，所有方法均为异步
- `base_url` 可配置，支持任何 OpenAI 兼容服务（如 DeepSeek、本地 Ollama 等）
- 模型名称可通过 `.env` 灵活切换

### 4.5 Agent 核心引擎

#### 4.5.1 调研编排器 — `backend/agent/orchestrator.py`

```python
# 文件：backend/agent/orchestrator.py
```

**职责：** 作为最高层控制器，编排整个调研流程的四个阶段。

**核心类：`ResearchOrchestrator`**

**`__init__()`**：创建 `LLMClient` 实例和 `ToolRegistry` 实例，通过 `_setup_tools()` 注册三个内置工具。

**`_setup_tools()`**：向注册中心注册三个工具：

| 工具名 | 描述 | 参数 | 处理函数 |
|--------|------|------|----------|
| `web_search` | 在互联网上搜索信息 | `query: string` | `web_search()` |
| `fetch_page` | 抓取并提取指定 URL 的文本内容 | `url: string` | `fetch_page()` |
| `cross_check` | 对已收集的事实进行交叉验证 | `facts: string` | `cross_check()` |

**`research(query) → AsyncGenerator[dict, None]`**：核心调研流程，是一个异步生成器，yield 的事件通过 SSE 实时推送。

**四阶段流程：**

1. **生成计划**：调用 `self.llm.generate_plan(query)`，yield `PlanEvent`
2. **ReAct 循环**：创建 `AgentLoop` 实例，`async for step in loop.run(query)`，逐步骤 yield `StepEvent`，同时收集所有 Observation 内容到 `all_observations`
3. **交叉验证**：取最近 5 条 Observation 拼接，调用 `cross_check(facts)`，yield `CrossCheckEvent`
4. **生成报告**：将问题、子问题、收集到的信息、交叉验证结果组合成 prompt，调用 `self.llm.chat()` 生成 Markdown 报告，通过正则提取所有 URL 作为来源列表，yield `ReportEvent`

**`_extract_sources(report) → list[str]`**：用正则 `https?://[^\s\)\]]+` 从报告中提取所有 URL，并用 `dict.fromkeys()` 去重保留顺序。

#### 4.5.2 ReAct 循环引擎 — `backend/agent/loop.py`

```python
# 文件：backend/agent/loop.py
```

**职责：** 实现 ReAct（Reasoning + Acting）范式的核心循环，是 Agent 的"大脑"。

**System Prompt（中文）：**

```
你是一个深度调研智能体。你的目标是对研究主题进行彻底调查，生成一份全面、来源可靠的答案。

每轮请按以下格式回复：

THOUGHT: <分析目前已知信息，是否充分？下一步深入调研的方向是什么？>

然后选择：
ACTION: <工具名>|<JSON参数>

或者当信息足够时：
FINAL_ANSWER: <用 Markdown 格式编写的综合调研报告，包含引用来源>

规则：
- 先搜索后阅读：先用 web_search 搜索，再对最有价值的 URL 使用 fetch_page 深入阅读
- 多源交叉验证：重要结论不能只依赖单一来源
- 生成最终答案前必须对事实进行交叉验证
- 引用来源格式：[来源: URL]
- 如果搜索失败或无结果，调整策略或诚实地报告未能找到的内容
```

**核心类：`AgentLoop`**

**`__init__(tool_registry, llm)`**：接收工具注册中心和 LLM 客户端，创建 `MemoryManager` 实例。

**`run(query) → AsyncGenerator[StepEvent, None]`**：ReAct 主循环。

**循环逻辑（`for round_num in range(1, MAX_ROUNDS + 1)`）：**

1. **LLM 推理**：调用 `self.llm.chat(messages)` 获取原始输出
2. **解析输出**：调用 `parse(raw_output)` 提取结构化指令
3. **发送 Thought**：如果解析出 Thought，创建 `thought` 类型的 StepEvent 并 yield
4. **解析失败处理**：如果 `parsed.parse_error and not parsed.is_final`，将错误信息作为 user message 追加到 messages 中，让 LLM 重试一次（`continue`）
5. **执行工具**：如果解析出 Action：
   - 发送 `action` 类型的 StepEvent
   - 调用 `self.tool_registry.execute(action, params)` 执行工具
   - 发送 `observation` 类型的 StepEvent（内容截断为 2000 字）
   - 将 assistant 输出和 Observation 追加到 messages 上下文
   - 如果 Observation 超过 200 字，提取前 300 字作为关键事实存入 MemoryManager
6. **终止判断**：如果 `parsed.is_final`，yield 最终答案并 `return`
7. **强制终止**：达到 MAX_ROUNDS 后，调用 LLM 基于 `memory.get_steps_for_context()` 生成摘要报告

**设计要点：**
- 每轮都将完整的 `messages` 历史发送给 LLM，保证上下文连贯
- Observation 内容截断为 2000 字，防止单轮上下文爆炸
- 只有 Observation 长度 > 200 时才提取关键事实（过滤掉无意义的短结果）
- 达到最大轮数时强制生成摘要，防止无限循环

#### 4.5.3 输出解析器 — `backend/agent/parser.py`

```python
# 文件：backend/agent/parser.py
```

**职责：** 从 LLM 的原始文本输出中提取结构化指令。LLM 的输出格式不可靠，解析器需要处理各种格式偏差。

**核心数据结构：`ParsedOutput`**

```python
@dataclass
class ParsedOutput:
    thought: str                    # 推理内容
    action: str | None = None       # 工具名
    action_params: dict | None = None  # 工具参数
    is_final: bool = False          # 是否为最终答案
    final_answer: str | None = None # 最终答案内容
    parse_error: str | None = None  # 解析错误信息
```

**`parse(raw) → ParsedOutput`** 解析逻辑：

1. **提取 THOUGHT**：正则 `THOUGHT:\s*(.+?)(?=\n(?:ACTION|FINAL_ANSWER)|\Z)`，使用 `re.DOTALL` 支持多行思考内容，`re.IGNORECASE` 忽略大小写。

2. **检测 FINAL_ANSWER**：检查是否存在 `FINAL_ANSWER` 关键字。如果存在：
   - `is_final = True`
   - 提取 FINAL_ANSWER 后的内容作为最终答案
   - 同时兼容 "FINAL_ANSWER:" 有冒号和 "FINAL_ANSWER" 无冒号两种情况

3. **提取 ACTION**：正则 `ACTION:\s*(.+?)(?=\n(?:THOUGHT|OBSERVATION|FINAL_ANSWER)|\Z)`。仅在非最终答案时处理。Action 格式为 `工具名|JSON参数`：
   - 按第一个 `|` 分割得到工具名和 JSON 字符串
   - 尝试 `json.loads()` 解析参数
   - 如果 JSON 解析失败，设置 `parse_error` 并返回空字典作为参数

4. **兜底处理**：如果 THOUGHT、ACTION、FINAL_ANSWER 全都没有解析到，设置 `parse_error` 并将原始文本前 500 字作为 thought。

**容错设计：**
- 大小写不敏感（`re.IGNORECASE`）
- 支持多行 THOUGHT（`re.DOTALL`）
- 兼容 FINAL_ANSWER 有/无冒号两种格式
- JSON 解析失败时记录错误但不崩溃
- 完全无法解析时取前 500 字作为 thought，由 loop.py 的 retry 机制处理

#### 4.5.4 记忆管理器 — `backend/agent/memory.py`

```python
# 文件：backend/agent/memory.py
```

**职责：** 管理 Agent 的步骤历史，在上下文过长时自动压缩，同时保留关键事实。

**核心类：`MemoryManager`**

**属性：**
| 属性 | 类型 | 说明 |
|------|------|------|
| `steps` | `list[Step]` | 完整的步骤历史记录 |
| `key_facts` | `list[str]` | 去重的关键事实列表 |
| `max_history_chars` | `int` | 压缩阈值，默认 8000 字符 |

**方法：**

**`add_step(step)`**：将步骤追加到 `steps` 列表。

**`add_key_fact(fact)`**：将关键事实追加到 `key_facts` 列表（自动去重）。在 `loop.py` 中，只有 Observation 长度 > 200 时才提取前 300 字作为关键事实。

**`get_steps_for_context() → str`**：构建用于发送给 LLM 的上下文字符串。

- **正常模式**：如果所有步骤的格式化字符串总长度 ≤ `max_history_chars`，直接拼接返回
- **压缩模式**：如果超出限制：
  1. 从后往前取最近的步骤，直到达到 `max_history_chars // 2`（保留最近 4000 字符）
  2. 对前面的旧步骤调用 `_build_summary()` 生成压缩摘要
  3. 返回格式：`[摘要] + --- Recent Steps --- + [最近步骤]`

**`_build_summary(early_steps)`**：生成压缩摘要，包含：
- 被压缩的步骤数量
- 早期执行的操作数量
- 所有已收集的关键事实列表

**设计要点：**
- 压缩策略：保留最近步骤的完整内容（因为最新信息最重要），旧步骤只保留摘要
- 关键事实独立于步骤存储，即使步骤被压缩，关键事实也不会丢失
- 500 字符的关键事实提取阈值有效过滤了空结果和错误信息

#### 4.5.6 Web 搜索工具 — `backend/agent/tools/web_search.py`

```python
# 文件：backend/agent/tools/web_search.py
```

**函数签名：** `async web_search(query: str, num: int = 5) → str`

**执行流程：**
1. 检查 `SEARCH_API_KEY` 是否配置，未配置则返回提示信息
2. 使用 `httpx.AsyncClient`（15 秒超时）向 SerpAPI 发送 GET 请求

**错误处理：**
- API Key 未配置：返回提示信息
- 请求超时（`httpx.TimeoutException`）：返回超时提示，建议使用更精确的搜索词
- 其他异常：返回失败信息和异常详情
- 无搜索结果：返回 `未找到与「xxx」相关的结果`

#### 4.5.7 网页抓取工具 — `backend/agent/tools/fetch_page.py`

```python
# 文件：backend/agent/tools/fetch_page.py
```

**函数签名：** `async fetch_page(url: str, max_chars: int = 5000) → str`

**执行流程：**
1. 使用 `httpx.AsyncClient`（20 秒超时，因为网页可能较大）发送 GET 请求
2. 设置 User-Agent 为 `Mozilla/5.0 (compatible; ResearchAgent/1.0)`，声明身份
3. 使用 BeautifulSoup 解析 HTML
4. **清洗 HTML**：删除 `script`、`style`、`nav`、`footer`、`header`、`aside` 标签（这些通常不是正文内容）
5. 使用 `get_text(separator="\n", strip=True)` 提取纯文本
6. 去除空行，将文本拼接
7. 如果超过 `max_chars`（5000 字），截断并标注 `[内容已截断，超过 5000 字符]`
8. 提取网页标题，返回格式：`标题：...\n链接：...\n\n{正文}`

**错误处理：**
- 超时：返回 `抓取 url 超时，将使用搜索摘要`
- 其他异常：返回 `抓取 url 失败：{异常}，将使用搜索摘要`

异常信息中的"将使用搜索摘要"提示 LLM 可以依赖已有的搜索摘要继续工作。

### 4.7 API 路由层 — `backend/routes/research.py`

```python
# 文件：backend/routes/research.py
```

**职责：** 定义所有对外 API 端点。

#### `POST /api/research` — 发起调研（SSE 流）

- 接收 `{"query": "..."}` 请求体
- 生成 UUID 作为 session_id，立即创建数据库记录（status='running'）
- 创建 `ResearchOrchestrator`，异步迭代 `orchestrator.research(query)` 返回的事件

**`_event_type(event)` 辅助函数：** 根据事件内容判断 SSE 事件类型：

| 条件 | event 类型 |

| 包含 `sub_questions` | `plan` |
| 包含 `consistency` | `cross_check` |
| 包含 `markdown` | `report` |
| type 为 thought/action/observation | `step` |
| 其他 | `message` |

返回最近 50 条会话的摘要信息。

#### `GET /api/history/{session_id}` — 获取会话详情

返回单条会话的完整信息，包括反序列化的 `steps` 数组（从 `steps_json` 解析）。

#### 4.8.2 SSE 流式接收 — `frontend/src/composables/useSSE.js`

## 六、容错与异常处理

| 场景 | 位置 | 处理策略 |
|------|------|----------|
| LLM 输出格式无法解析 | `loop.py` + `parser.py` | 将格式错误信息反馈给 LLM，给予一次重试机会 |
| Search API 超时/限流 | `web_search.py` | 返回友好提示，LLM 可基于已有信息继续或调整搜索词 |
| 网页抓取失败 | `fetch_page.py` | 降级为"使用搜索摘要"，标注未能获取全文 |
| LLM 持续不输出 FINAL_ANSWER | `loop.py` | 硬上限 10 轮后强制调用 LLM 生成摘要报告 |
| 上下文超限（>8000字） | `memory.py` | 压缩早期步骤为摘要，保留最近步骤完整内容 |
| 用户关闭浏览器/断开 SSE | `routes/research.py` | 已完成步骤已实时写入数据库（收集后统一更新） |
| 工具执行异常 | `registry.py` | 捕获所有异常并返回字符串错误信息，不抛异常 |
| JSON 解析失败（plan/验证） | `client.py` / `cross_check.py` | 降级为默认值 |
| 数据库连接异常 | `connection.py` | 应用关闭时通过 lifespan 确保连接正确关闭 |

## 八、关键设计决策

1. **不使用 Agent 框架**：LangChain/LlamaIndex 增加了抽象层级和依赖复杂度，对于 ReAct 这种简单范式，自研约 500 行代码即可完整实现，且完全可控。

2. **SSE 而非 WebSocket**：调研场景是单向的服务器→客户端推送，SSE 比 WebSocket 更轻量，浏览器原生支持自动重连，且兼容 HTTP/2。

3. **Fetch API 而非 EventSource**：因为需要 POST 请求发送查询内容，EventSource 仅支持 GET，所以使用 Fetch API + ReadableStream 手动解析 SSE 流。

4. **低温度参数 (0.3)**：调研任务需要事实准确性而非创意性，低温度减少 LLM 的随机性，提高输出的一致性和可靠性。

5. **先压缩后截断**：MemoryManager 不是简单的字符截断，而是先压缩旧步骤为摘要再拼接最近步骤，保留最重要的上下文信息。

6. **所有错误都返回字符串**：Tool 执行失败时不抛异常，而是返回字符串错误信息，让 LLM 可以"读到"错误并据此调整策略，这是 ReAct 范式的核心优势。

7. **WAL 模式 SQLite**：Write-Ahead Logging 允许多个读操作与一个写操作并发执行，对 SSE 流式写入场景友好。
